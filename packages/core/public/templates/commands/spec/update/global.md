# /lattice/spec/update/global

**[执行前必读]** 执行本命令前必须先用 Skill 工具调用 `lattice` skill，再继续后续步骤。

**[依赖 skill 子文档]**（本命令期间会反复 read 的 skill 子文档）：
- `spec-workflows.md`：spec 定义 / 层级（含 user / global 必须附 `## 适用范围`）/ 沉淀判定 / 写入流程
- `SKILL.md`：索引维护（`ltc rag update`）
- `lattice-rules.md` §七：spec 优先级与冲突

**目标**：把对多用户、多项目都成立的高层规则沉淀到全局 spec。

## 什么时候使用

判定核心问题：**这条规则已经不是某个项目或某个用户的偏好，具备明显的通用性和稳定性，作为全局默认规则能减少重复沟通**。

spec 双重职能详见 skill `spec-workflows.md` 的#spec 定义。全局 spec 只保留**高价值、低争议、可迁移**的内容：

- **通用约束类**：跨用户跨项目都成立的命名原则、文档结构通则、任务管理习惯、通用协作规则
- **通用认知 / 方法论类**：跨项目通识（通用的代码组织心智、命名直觉）、适用于多用户多项目的标准方法论、通用的排错或交付节奏

完整沉淀判定见 skill `spec-workflows.md` 的#沉淀判定。

## 执行步骤

**统一遵循 skill `spec-workflows.md` 的#写入流程 的六步流程**。

### 全局级特有要求

1. **必须声明 `## 适用范围`**：全局 spec 影响所有用户所有项目，必须明确适用边界，避免被盲目套用。详见 skill `spec-workflows.md` 的#层级（中关于 user / global 必附 `## 适用范围` 的要求）
2. **写入原则**：
   - 全局 spec 只保留高价值、低争议、可迁移的规则
   - 适合放全局的通常是：命名原则、文档结构、任务管理习惯、通用协作规则
   - **不要把技术栈特定规则或团队私有约定写到全局**
3. **冲突检测**：全局规则是默认值，但下层（user / project）可能已有更具体规则。读取下层内容确认新规则不会与下层产生混淆；如果下层已有更严格特例，确认全局规则是否作为合理默认值

层级判定详见 skill `spec-workflows.md` 的#层级。

## 输出要求

- **说明为什么这条规则适合全局级**
- 说明更新了哪个全局 spec 文件
- 检测到与下层 spec 的语义冲突时明确告知：冲突点 / 建议处理方式
- 提醒用户项目级和用户级规则仍然可以覆盖全局默认规则
