# /lattice/spec/update/user

> **[执行前必读]** 执行本命令前，必须先使用 Skill 工具调用 `lattice` skill，阅读完整的 Lattice 使用说明，再继续执行后续步骤。

目标：把跨多个项目可复用、但仍属于当前用户工作方式的经验沉淀到用户级 spec。

## 什么时候使用

- 这是你个人或团队在多个项目中重复采用的做法
- 它不应只属于当前项目
- 它又没有普适到“所有用户、所有项目”都适合

## 执行步骤

1. 先确认这条规则不是项目特例：

```bash
lattice context
lattice search "<相关主题>"
```

2. 查看当前已有用户级 spec：

```bash
lattice spec list --scope user
```

3. 如果是**新建** spec，先检查是否有可参考的模板结构：

```bash
lattice spec template list
```

如果有匹配的模板，参考其文件组织和内容结构，保持跨项目的一致性。

4. 更新或新建用户级 spec 文件
5. **冲突检测**：写入前读取全局级和已有项目级 spec 的内容，判断是否存在语义矛盾：

```bash
lattice spec list --scope global
lattice spec show <相关文件>
```

不仅检查同名文件，更要读取内容判断规则是否互斥。如果新规则与全局级不一致，要说明这是刻意覆盖还是应合并。

## 写入原则

- 重点沉淀可复用的工作流、代码风格、架构偏好、协作方式
- 尽量避免把技术栈强绑定到某个具体项目
- 如与项目级 spec 冲突，不要强行抹平，让项目级保留特例

## 输出要求

- 明确说明这条规则为什么适合用户级而不是项目级
- 说明更新了哪个文件
- 如检测到语义冲突，必须明确告知用户：冲突点、影响范围、建议处理方式
- 提醒用户项目级 spec 仍然可以覆盖该规则
