# /lattice/spec/sediment

> **[执行前必读]** 执行本命令前必须先用 Skill 工具调用 `lattice` skill，再继续后续步骤。
>
> **[依赖 skill 子文档]**（本命令期间会反复 read 的 skill 子文档）：
> - `spec-workflows.md`：spec 是什么 / 层级判定 / 沉淀判定 / 写入流程
> - `SKILL.md`：索引维护（`ltc rag update`）
> - `lattice-rules.md` §七：spec 优先级与冲突

**目标**：AI 自主扫描当前会话中的可沉淀内容，自行判定每条应放在哪个层级（项目/用户/全局），向用户确认后执行写入。用户无需手动指定层级。

## 与 spec/update/{project|user|global} 的区别

| | `/lattice/spec/update/<层级>` | `/lattice/spec/sediment` |
|---|---|---|
| 层级由谁决定 | 用户指定 | AI 自主判定 |
| 沉淀内容由谁识别 | 用户指定或暗示 | AI 从对话/任务中提取 |
| 适用场景 | 用户明确知道要沉淀什么、放在哪 | 对话结束/任务归档前，批量沉淀 |

## 执行步骤

### 步骤 1：识别可沉淀内容

回顾当前会话（含 PRD、checkpoint、design.md、对话历史），按 skill `spec-workflows.md`「沉淀判定」的三档标准筛选：

- **必须写**：用户显式指示行为规则 / 用户主动给出项目认知
- **建议写**：满足核心判定（下次还需要这条信息吗？）
- **不写**：一次性需求 / 未验证猜测 / 纯任务级细节

跳过"不写"类，保留"必须写"和"建议写"类作为候选。

### 步骤 2：自主判定层级

对每条候选内容，按以下判定树分派层级：

```
该信息只对当前项目有效？
  ├─ 是 → 项目级
  └─ 否 → 跨项目可复用但属于个人/团队偏好？
           ├─ 是 → 用户级
           └─ 否 → 多用户多项目通用且低争议？
                    ├─ 是 → 全局级
                    └─ 否 → 用户级（存疑默认用户级）
```

判定依据参照 skill `spec-workflows.md`「层级」：
- 项目级：只对当前项目有效（架构、目录、领域概念、项目特有流程）
- 用户级：跨项目可复用的个人规范/方法论/经验
- 全局级：高价值、低争议、可迁移的通用默认规则（极少使用）

### 步骤 3：向用户确认沉淀计划

输出沉淀计划表，格式：

```
## 沉淀计划

| # | 内容摘要 | 层级 | 目标文件 | 判定理由（一句话） |
|---|---|---|---|---|
| 1 | ... | 项目级 | xxx.md | 只对当前项目结构有效 |
| 2 | ... | 用户级 | yyy.md | 跨项目通用的排查方法论 |

是否确认执行？可修改层级或排除某项。
```

等待用户确认。用户可以：
- 直接确认
- 调整某项的层级
- 排除某项
- 补充遗漏项

### 步骤 4：执行写入

用户确认后，对每条内容统一遵循 skill `spec-workflows.md`「写入流程」的六步流程执行：

1. 前置信息采集（`ltc spec list` 查已有 spec）
2. 检查模板与已有 spec（优先补充而非重复创建）
3. 冲突检测（读取同层级及上下层级已有 spec）
4. 写入（项目级不需要 `## 适用范围`；用户级/全局级必须附 `## 适用范围`）
5. 元数据刷新（`ltc spec migrate`）
6. 索引更新（`ltc rag update`）

多条 spec 可批量写入，但每条独立做冲突检测。

## 命令参数

- 无参数：扫描整个当前会话
- `--task <id>`：扫描指定任务的 PRD + progress + design.md
- 后跟描述文本：仅对用户指定的内容做层级判定和沉淀（不做全量扫描）

## 输出要求

- 步骤 3 的确认表必须输出，不得跳过确认直接写入
- 写入完成后汇报：更新/新建了哪些 spec 文件 + 各自层级
- 检测到冲突时在确认表中标注，由用户决定
