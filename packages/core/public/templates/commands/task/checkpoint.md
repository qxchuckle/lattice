# /lattice/task/checkpoint

目标：记录当前任务的关键进展，确保过程信息结构化落盘，支持跨会话追踪。

## 如何理解命令参数

- 如果命令后有任务 ID：直接对该任务添加检查点
- 如果命令后没有任务 ID：自动使用当前上下文中正在进行的任务（通过 `lattice task list --current --status in_progress` 推断）

## 执行步骤

### 1. 确定目标任务

如果用户提供了任务 ID：

```bash
lattice task info <task-id>
```

如果未提供任务 ID，先推断当前任务：

```bash
lattice task list --current
```

选择状态为 `in_progress` 的任务。如果有多个，询问用户确认。

### 2. 总结当前会话进展

结合当前对话内容，归纳出：

- **类型（type）**：从以下枚举中选择最合适的一个
  - `decision`：做出了设计/技术决策
  - `issue`：发现了问题或踩坑
  - `pivot`：方案发生重大调整
  - `summary`：会话摘要或阶段总结
  - `milestone`：里程碑达成
  - `note`：一般记录
- **标题（title）**：一句话概括，简洁明确
- **内容（message）**：补充说明，包括原因、影响、后续计划等

### 3. 写入检查点

```bash
lattice task checkpoint <task-id> --type <type> --title "<标题>" -m "<内容>"
```

## 隐式触发时机

Agent 应在以下时机主动执行 checkpoint，无需用户显式调用：

- 用户确认了一个设计决策或技术选型
- 发现了意料外的问题、Bug 或兼容性坑
- 方案从 A 调整为 B（pivot）
- 一个阶段性目标完成（某模块实现完毕、测试通过等）
- 会话即将结束，或用户表示"先到这"、"下次继续"
- 用户反馈了一个重要约束或修正

## 输出要求

- 告诉用户检查点已记录
- 显示检查点 ID、类型、标题
- 如果是隐式触发，简要说明为什么在此时记录

## 注意事项

- 不要在每次对话轮都记录，只在有实质性进展时记录
- 标题应简洁（不超过 30 字），message 可以详细
- 如果一次对话中有多个值得记录的进展，可以分多次调用
- 检查点记录的是过程信息（决策、问题、调整），不是收敛型内容（目标、约束）——后者应更新到 prd.md
