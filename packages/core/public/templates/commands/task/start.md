# /lattice/task/start

> **[执行前必读]** 执行本命令前，必须先使用 Skill 工具调用 `lattice` skill，阅读完整的 Lattice 使用说明，再继续执行后续步骤。

目标：开始一个任务，并让当前会话和任务状态保持一致。

## 如何理解命令参数

- 如果命令后是一个已存在的任务 ID：将该任务置为 `in_progress`
- 如果命令后是非 ID 的描述、关键词、文件引用或一段需求：先结合命令参数和当前对话，总结出一个简洁、明确的任务标题；不要直接把原始命令参数当作任务标题。再检查当前项目里是否已有相似且正在进行中的任务；若有，先提醒用户确认是否继续该任务，否则再新建任务并开始

## 执行步骤

### 情况一：参数是任务 ID

运行：

```bash
lattice task start <task-id>
lattice context --task <task-id>
```

### 情况二：参数不是任务 ID

先做一步归纳：

- 结合命令参数和当前对话，提炼出一个更适合作为任务名称的标题
- 不要直接复用原始命令参数，尤其不要把文件路径、文件引用、整段需求描述或命令噪音原样当成标题
- 如果命令参数只提供了局部线索，允许结合当前会话主题补全标题

然后，如果当前目录是已注册项目，运行：

```bash
lattice task list --current
lattice search "<总结出的任务标题>" --project <project-id> --type task
```

先判断：

- 当前项目里是否已有标题相近、语义相近，且状态为 `in_progress` 的任务
- 如果有，先把这些任务列给用户，提醒用户是否其实是要继续其中某个已有任务
- 如果这是某个既有任务的明确后续步骤、拆分步骤或子任务，创建时应补上父任务关系
- 只有在没有发现相似进行中任务，或用户明确表示需要新建任务时，才继续运行：

```bash
lattice task create "<总结出的任务标题>" --current
```

如果已经能确认父任务，再改为：

```bash
lattice task create "<总结出的任务标题>" --current --parent <parent-task-id>
```

拿到任务 ID 后，再运行：

```bash
lattice task start <task-id>
lattice context --task <task-id>
```

## 开始任务后

### 参考近似任务

任务开始后，应主动搜索已完成的近似任务作为参考。搜索方式：

```bash
lattice search "<当前任务标题或核心关键词>" --type task --json
```

根据搜索结果和当前任务的复杂性，自行判断需要参考多少个近似任务的 PRD：

- **简单/常规任务**（模式清晰、涉及范围小）：参考 1 个最相关的已完成任务 PRD 即可
- **中等复杂任务**（涉及多个模块或需要理解上下文模式）：参考 2~3 个相关任务 PRD
- **高复杂任务**（架构级变更、跨模块协作、首次涉足的领域）：参考 3~5 个相关任务 PRD，必要时还应查看它们的 `design.md` 和 `progress.yaml`

判断依据：
- 搜索结果中 score 较高且与当前任务领域/模式高度相关的优先参考
- 如果当前任务是某类模式的重复应用（如"适配 X 能力"），应参考同类已完成任务了解标准做法
- 如果当前任务涉及此前未触及的架构或概念，应广泛参考多个相关任务建立认知

不要机械地固定只看 1 个任务。参考数量应与任务复杂性匹配。

### 读取已有信息

- 如果任务目录下已有 `design.md`（说明之前做过方案讨论），先 read_file 读取其内容，了解已讨论的方案、约束和结论，避免重复讨论或偏离已有决策。
- 任务开始后，应尽快完善该任务目录下的 `prd.md`，不要只保留创建时的空白标题。
- PRD 只记录收敛型内容：目标、约束、关键设计思路、最终方案概览、文件索引。
- `prd.md` 可以只是这个任务目录的主入口，不要求把所有内容都写在一个文件里。
- 如果任务太大、单个 `prd.md` 已经过长，或者任务本身分成多个步骤、多个阶段或 plan 形式推进，可以把详细内容拆到该任务目录下的其他 Markdown 文件中，再由 `prd.md` 负责概览、索引和跳转。
- 如果后续对话中用户补充了需求、边界、设计方案、取舍理由或实现计划，应自行判断这些信息是否会影响任务理解；如果会，就同步更新 `prd.md`。
- 如果在任务执行过程中发现实际涉及的项目范围发生了变化，例如新增了其他关联项目，或确认某些项目不再相关，应同步更新该任务元数据里的 `projects` 字段，不要让任务关联项目与实际工作范围脱节。
- 如果 `prd.md` 过长，可以把详细设计、方案对比、实施计划等拆到其他 Markdown 文件中做渐进式加载；但 `prd.md` 仍然必须是该任务的必要入口，负责概览、索引和跳转关系。
- 当设计发生明显变化时，优先更新 PRD，再继续执行实现或分析，避免任务上下文和实际方案脱节。

## 任务进展追踪

任务执行过程中，应主动通过 `lattice task checkpoint` 命令记录关键进展，确保过程信息不丢失：

```bash
lattice task checkpoint <task-id> --type <type> --title "<标题>" -m "<内容>"
```

可用类型：`decision` / `issue` / `pivot` / `summary` / `milestone` / `note`

建议在以下时机主动记录：

- 做出了关键设计/技术决策（type: decision）
- 发现了意料外的问题或坑（type: issue）
- 方案发生了重大调整（type: pivot）
- 一个阶段性里程碑完成（type: milestone）
- 会话即将结束或用户明确表示“先到这”（type: summary）
- 用户手动要求记录当前状态（type: summary 或 note）

查看已记录的进展：

```bash
lattice task progress <task-id>            # 全部
lattice task progress <task-id> --last 3   # 最近 3 条
lattice task progress <task-id> --type decision  # 只看决策
```

## 输出要求

整体原则：**精简但不静默**。要求精简是指不要长篇复述 CLI 输出、不要罗列每一步命令；但在关键阶段必须给出简短说明，让用户随时能确认任务状态，不能一路静默地连续执行 CLI 命令。

### 阶段性输出（必须）

在执行过程中，到达以下关键节点时，必须立即输出一段简短说明（一般 2~5 行），不要等所有步骤都跑完再统一汇报：

- **任务创建完成后**：立即说明
  - 最终采用的任务标题（特别是从非 ID 参数归纳出来的情况）
  - 新任务 ID
  - 父任务 ID（如有）
  - 创建时关联的项目（项目名 + ID 简写即可）
- **任务启动 (`task start`) 完成后**：说明任务当前状态已置为 `in_progress`
- **关联项目发生变化时**（例如执行了 `task associate`）：说明当前最新的关联项目列表
- **发现相似进行中任务时**：先停下来把候选任务列给用户确认，不要直接新建

### 任务进入实施前的最终确认

在开始具体实施前，再补一段简短的整体确认输出：

- 任务 ID、当前状态
- 任务标题、关联项目、父任务（如有）
- 提炼出的最关键上下文与约束（不超过几条要点）

### 其他要求

- 如果任务执行中更新了关联项目，明确告诉用户 `projects` 字段已同步更新以及当前关联了哪些项目
- 开始任务后主动整理并维护该任务的 `prd.md`；如果 PRD 被拆分，仍保持 `prd.md` 作为任务主入口
- 如果当前目录不是已注册项目，明确提示无法自动关联当前项目
