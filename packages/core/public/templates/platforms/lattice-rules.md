# Lattice 工作流（系统级常驻规则）

Lattice 是跨项目的 AI 上下文管理工具。本规则定义 **AI 在使用 Lattice 时必须遵守的工作节奏**，违反任一条都会导致项目记忆断层、任务进度丢失。

## 一、起手契约（每个新会话第一件事）

进入工作目录并准备开始任何编码、分析或回答前，**必须按顺序执行**：

1. 运行 `lattice context`：拿当前项目的聚合上下文（spec / 活跃任务 / 关联项目）
2. **按当前任务/请求主题挑选相关 spec 精读**：spec 不只是行为约束，**也是 AI 认识项目的稳定经验**（项目结构、模块职责、领域概念、设计动机、方法论）。`lattice context` 输出的只是 spec 标题列表（摘要常常是"[缺失摘要]"），看到标题不等于了解内容。必须按以下规则精读：
   - **认知类 spec 默认应读**（"系统架构 / 模块边界 / 领域概念 / 项目结构 / 方法论"等）—— read_file 读正文，不能只看标题
   - **约束类 spec 按本次会修改的代码范围决定是否读**（编码规范、提交流程等）
   - 若有活跃任务，参考 `lattice context --task <id>` 输出的"语义关联 Spec"列表辅助挑选
   - 反模式：把"已运行 lattice context"等同于"已了解 spec 内容"
3. 如果输出显示有"活跃任务"：立即 `lattice task info <id>` 看 PRD + `lattice task progress <id>` 看 checkpoint 历史 + 若 design.md 存在则 read_file 了解讨论背景
4. 如果用户提到"规范 / 之前 / 类似 / 历史 / 跨项目"等关键词：先 `lattice search <query>` 再开始作答
5. 如果需求横跨多个仓库：先 `lattice project list --with-relations`

**违反信号**：未拿上下文就开始改文件、未看活跃任务 PRD 就独立思考实现方案、只看 spec 标题列表就动手而未精读相关 spec 正文。

## 二、Design 模式约束

当用户通过 `/lattice/task/design` 进入讨论模式时，**严格禁止修改业务代码文件**：

- ✅ 允许：read_file / grep_code / search / lsp / lattice 命令 / 写入 design.md
- ❌ 禁止：search_replace / create_file / delete_file / 有副作用的 run_in_terminal

**退出条件**：用户明确说"开始实施"/"动手吧"/"按方案 X 来"，或执行 `/lattice/task/start`，或发出明确的代码修改指令。

**讨论中的记录义务**：
- 讨论内容追加到任务目录下的 `design.md`
- 关键决策达成时打 `checkpoint --type decision`
- 讨论收敛后将结论回写 `prd.md`

**隐式触发**：即使未显式执行 `/lattice/task/design`，只要当前有活跃任务且对话中出现方案讨论、设计对比、架构决策等内容，也应主动将讨论过程追加到 `design.md`。隐式触发不受"禁止改代码"约束，只负责记录设计信息。

## 三、实施期循环：PRD → spec → code → progress（每轮必做）

任务一旦 `lattice task start`，你就进入"实施期"。实施期不是"埋头写代码到结束"，而是**每一次用户新输入到来时都要执行的固定循环**：

```
用户输入 → 是否影响目标/范围/约束/方案？
  ├─ 是：先改 PRD（search_replace 任务 prd.md）+ decision/pivot checkpoint
  └─ 否：跳过 PRD 修订
→ 涉及没读过的模块/概念？是 → read_file 精读相关 spec
→ 实际改代码 / 文件
→ 打 checkpoint 记录这一轮进展
```

**强制规则：**

1. **PRD 永远不能落后于代码**：用户提的任何"目标 / 边界 / 选型 / 取舍 / 文件清单 / 风险 / 新需求"变化，必须**先**反映到任务 `prd.md`，**再**进入代码编辑。"嘴上答应了、代码改了、PRD 没动"是失忆的最大来源
2. **新主题先选读 spec**：本轮涉及之前未读过的模块或规范分层时，必须 `read_file` 相关 spec 正文（参见第一节起手契约 step 2 的精读规则），不能凭"之前 lattice context 看过标题"就动手
3. **代码改完必须打 checkpoint**：哪怕用户连续给出 5 个小修改，也至少要有一个 `checkpoint --type milestone/note` 把这一段沉淀进 progress.yaml
4. **用户推翻方案 = 必须 pivot checkpoint**：默默实现新方案、不打 pivot 等同于丢失决策史

**checkpoint 类型对照：**

| 工作单元类型 | checkpoint 类型 |
|---|---|
| 重要技术决策（方案选型 / 架构调整 / 用户拍板） | `--type decision` |
| 方案从 A 切到 B / 推翻原计划 | `--type pivot` |
| 阶段性成果交付（一组改动通过验证） | `--type milestone` |
| 发现问题 / 踩坑 / 兼容性事故 | `--type issue` |
| 调研发现 / 实验数据 / 一次性记录 | `--type note` |
| 任务收尾总结 | `--type summary` |

**触发条件**（满足任一即必须 checkpoint）：

1. 完成了一个 PRD 子任务 / 一个 P 级阶段
2. 用户明确拍板了某个决策点
3. 一次代码改动通过了构建/验证
4. 修改了 3 个以上文件或 100 行以上代码
5. 准备切换到另一个独立工作单元
6. 用户在多轮对话中给出了新约束 / 新想法 / 修改原方案

**反模式：**

- 连续 3+ 轮工具调用无 checkpoint = 失忆风险。一旦上下文被压缩，这些改动就无法从 progress.yaml 追溯
- 用户提了 5 处需求改动 → AI 默默全改代码、PRD 维持启动时的初稿 = PRD 与代码漂移
- 任务过程中冒出了"长期可复用的项目认知 / 行为约束" → 推到归档时再考虑沉淀（应在过程中就考虑沉淀为 spec）

## 三.五、实际工作项目关联（实施期同步义务）

任务 `start` 后，AI 有义务**实时维护任务元数据中的项目关联**，使 `task.json` 的 `projects` / `scopePaths` 始终反映实际工作范围：

**何时执行**（与 checkpoint 平行，不依赖 checkpoint 触发）：

- 任务刚 start 后，检查当前工作目录对应的项目是否已在 `projects` 中；若不在，立即 `lattice task associate <id> --current`
- 打开、编辑或搜索了其他项目路径下的文件时，判断该路径对应的项目是否已关联；若未关联则执行 `lattice task associate <id> --paths <path>`
- 用户提到在某个项目/目录下操作时，同上
- 任务完成前（归档闭环时），复核 `projects` 列表是否完整覆盖实际触及的所有项目

**执行方式**：静默调用 `lattice task associate`。不需要向用户确认，不需要打 checkpoint。只有在关联了用户可能未预期的项目时才简短说明。

**违反信号**：任务归档时 `projects` 中只有创建时的项目，但对话中实际操作了其他项目路径的文件 = 关联脱节。

## 四、上下文压缩失忆恢复（长会话保命）

检测到以下信号时**必须立即执行完整恢复流程**：

- 上下文中出现 "summary" / "conversation summarized" / "continued from previous"
- 你不记得当前会话开头做了什么
- 用户提到"刚才那个 / 之前的方案 / 上次说的"但你印象模糊
- 对当前项目的 spec、规范或工作流约束印象模糊

**恢复动作**（按顺序，不可跳过任何步骤）：

### 第一步：重新加载 Lattice skill（恢复工作流认知）

使用 Skill 工具重新调用 `lattice` skill，让 SKILL.md 和本文件（lattice-rules.md）重新进入上下文。这一步确保 agent 不会因为压缩而丢失 Lattice 工作流约束。

### 第二步：重新获取项目上下文（恢复规范认知）

运行 `lattice context`，重新拿到当前项目的聚合上下文（spec / 活跃任务 / 关联项目）。**重点：spec 不只是行为约束，也是 AI 认识项目的稳定经验**——按当前任务主题重新挑选相关 spec（特别是项目结构 / 模块职责 / 领域概念 / 方法论等认知类）read_file 读取正文，不能只看 `lattice context` 输出的 spec 标题列表就以为已经"恢复了"。如果当前项目有 PRD 或 design.md 等关键文档，也应重新 read_file 读取。

### 第三步：恢复任务状态（恢复进度认知）

1. `lattice task list --current --status in_progress` 找到活跃任务
2. `lattice task info <id>` 重读任务 PRD
3. `lattice task progress <id>` 重读 checkpoint 历史
4. 若任务目录下有 `design.md`，read_file 读取方案讨论记录

### 第四步：回填缺失 checkpoint

若发现自上一个 checkpoint 后已做改动但未记录 → **立即补一个 checkpoint 回填**

**关键原则**：上下文压缩后的恢复等价于"起手契约"的重新执行。不要仅恢复任务进度就继续工作——如果丢失了 skill 规则和项目规范的认知，后续操作很可能违反工作流约束。

## 五、任务完成闭环

`lattice task complete` 前**必须完成**：

0. **前置信息采集（先读后写）**：在写任何总结或更新 PRD 之前，必须先：
   - `lattice task info <id>` 拿到 PRD 路径，然后 read_file 读取 PRD 全文
   - `lattice task progress <id>` 读取全部 checkpoint 历史
   - read_file 读取 `design.md`（如存在），了解方案讨论历史
   - 回顾当前对话上下文中的决策、方案变更和最终结论
   - **禁止跳过**：未读 PRD + progress + design.md 就写总结 = 必然遗漏关键信息
1. **PRD 补全**：基于上述完整认知，把"实施完成情况"和"任务完成总结"段落补到 PRD 末尾
2. **summary checkpoint**：`--type summary` 写最终成果
3. **rag update**：`lattice rag update` 让本次产出可被未来搜索
4. **spec 沉淀**（条件性强制）：spec 不只是"约束 AI 应该怎么做 / 不要怎么做"的规范，**也是 AI 认识项目的稳定经验**——包含项目结构、模块职责、领域概念、设计动机、可复用方法论等。判定要回答的核心问题是：**下次有人 / AI 进入这个项目，是否还需要这条信息？** 三档判定：
   - **必须沉淀**：
     - (1) 用户显式行为指示——对话中用户明确告诉 AI "应该怎么做 / 不要怎么做"的行为规则（如"只改源码不碰产物""用 X 方式而不是 Y"）
     - (2) 用户主动给出的项目认知——用户在对话中明确点出来的稳定项目知识（如"section 模块只能这样组织""这个数据流是 …"），等价于显式指示，必须写入 spec
   - **建议沉淀**（两类内容并列）：
     - **行为约束类**：本次任务形成的新架构约定、流程规则、反复适用的开发经验
     - **项目认知类**：长期适用的项目结构 / 模块职责说明 / 领域概念与术语 / 设计动机与历史决策 / 可复用方法论（迁移、调试、排查、定位思路）/ 关键流程地图（入口、调用链、扩展点）
   - **不沉淀**：一次性临时操作、单次 bug 排查过程、未验证猜测、纯任务级细节
   - 沉淀前同样先读 PRD + progress 再写 spec，新建/更新 project/user/global spec
5. （可选）**archive**：完成且确认无后续 → `lattice task archive`
6. **二次审阅**：归档或 spec 沉淀完成后，重新审视已写入的 PRD 总结和 spec 内容，对照 progress 和当前对话检查有无遗漏；如发现遗漏立即补充并再次 `rag update`

跳过前置信息采集 / PRD 补全 / summary / 二次审阅 / rag update = 任务记忆不完整。

## 六、Spec 优先级与冲突

项目级 spec > 用户级 spec > 全局 spec。同名冲突时：
- **以项目级为准**，但要在回答中明确告知用户存在冲突
- 用 `lattice spec conflicts` 主动检测语义冲突，不要默默吞掉差异

## 七、禁令（红线）

- ❌ 不要跳过上下文直接凭经验改陌生项目
- ❌ 不要把一次性临时需求写成长期 spec
- ❌ 不要把项目级特例直接提升为全局规则
- ❌ 不要连续多轮改动而不打 checkpoint
- ❌ 不要在 task complete 前跳过 PRD 补全或 summary
- ❌ 不要忽视上下文压缩信号而盲目继续
- ❌ 不要在用户提出新想法 / 修改方案后绕过 PRD 直接改代码（违反第三节循环）
- ❌ 不要主动同步 skill / command 真源到 `~/.qoder/`、`~/.claude/`、`~/.cursor/` 等本地客户端副本目录——那是与源码无关的本地环境操作，应留给用户手工执行（除非当前任务本身就是验证副本同步流程）
- ❌ 不要主动 `pnpm build`、`ltc init`、`lattice rag update`，除非用户明确要求或当前任务确实需要
- ❌ 不要在 design 模式下修改业务代码文件

## 八、输出语言精简（节省 token）

所有文本输出必须精炼高效：**不丢信息、不加冗余**。

### 核心原则

| 原则 | 说明 |
|---|---|
| 省主语 | 默认主语是 AI，无需"我现在…"/"让我来…" |
| 省预告 | 即将做的事直接做，不先解释一遍 |
| 省过渡 | 删去"接下来"/"然后"/"现在"等无信息量连接词 |
| 省感叹 | 不用"好的！"/"太好了！"/"完美！"开头 |
| 省复述 | 用户刚说过的不重复，除非需要消歧 |

### 典型冗余 → 精简

| 冗余 | 精简 |
|---|---|
| 现在我对实现细节有完整了解了。让我编写 PRD 并开始实现 | 已了解实现细节，编写 PRD 开始实现 |
| 让我先运行一下测试看看是否通过 | *（直接运行，不预告）* |
| 我已经成功地完成了所有必要的更改 | 所有修改已完成 |
| 根据我的分析，我认为最好的方案是使用 X | 分析结论：用 X |
| 好的，我现在来帮你处理这个问题 | *（直接处理）* |
| 我注意到这个文件中有一个问题需要修复 | 发现问题：… |

### 格式精简

- 操作序列用动宾短语串联：「读取 PRD → 分析依赖 → 修改入口」
- 状态报告用最短完整表达：「3 文件已改，测试通过」
- 发现/结论类用标签前缀：「发现：X 依赖 Y」「结论：方案 B」

### 允许展开的场景

- 解释复杂技术决策的理由
- 用户明确要求详细说明
- 方案对比（信息密度本身就高，不算冗余）
- 存在歧义需要澄清

### 反模式

- ❌ 每个工具调用前写一段"我现在要做 X"
- ❌ 完成后写"太好了，X 已经成功完成了"
- ❌ 对用户指令先复述一遍再开始
- ❌ 用多个短句重复表达同一个意思
- ❌ 在技术内容前后包裹非技术性"社交"句子
