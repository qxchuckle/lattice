# Spec 工作流

本文件是 Lattice 中所有 **spec 概念与流程**的权威源。其他文档应通过锚点引用本文件，不再重复定义。

## spec 是什么

spec 记录一切对理解项目、完成任务有益、可复用参考、有价值的内容。

核心判定：**下次有人 / AI 进入这个项目，是否还需要这条信息？** 需要就写进 spec。

### 内容范围

spec 不仅是"不要做什么 / 应该做什么"。以下都属于 spec 应记录的内容：

- **行为约束**：编码规范、提交流程、架构边界、依赖方向、技术栈禁令
- **项目认知**：系统架构、模块职责、目录结构、领域概念、设计动机、关键流程地图
- **流程范式**：完成某类任务（迁移、重构、接入、调试）的完整步骤范式
- **经验细节**：具体的 API 映射、字段转换、配置开关、边界条件处理
- **试错积累**：哪些方案试过不行、为什么不行、最终怎么解决、失败方案的否决理由

> 抽象规则谁都能写。实战中踩过的坑、具体的操作步骤、失败方案的记录——这些才是最宝贵的、不可替代的。

### spec 不是永远正确的

项目演进、代码重构、需求变更都可能导致 spec 过时。发现 spec 与实际代码矛盾时：

1. 不要盲目服从 spec——结合实际代码做交叉验证
2. 必要时询问用户："规范中写的是 X，但实际代码是 Y，以哪个为准？"
3. 确认后及时更新 spec（`/lattice/spec/update/*`）或修复代码

## 层级

```
项目级 > [父项目级 > 祖先项目级 >] 用户级 > 全局级
```

| 层级 | 路径 | 适用 |
|---|---|---|
| 项目级 | `~/.lattice/users/<user>/projects/<id>/spec/` | 只对当前项目有效 |
| 用户级 | `~/.lattice/users/<user>/spec/` | 跨项目可复用的团队/个人习惯 |
| 全局级 | `~/.lattice/spec/` | 多用户多项目通用 |

冲突处理：近覆盖远、项目级最优先；同名冲突必须告知用户覆盖关系。

**user / global 必须包含 `## 适用范围`**（项目级不需要），声明适用于哪些项目/语言/框架，避免 AI 盲目应用到不相关项目。

## 读 spec

### 何时读

- 任务起手（`lattice context` 拿到 spec 列表后、动手前）
- 引入新模块 / 概念时
- 上下文压缩失忆恢复后

### 如何选

- 标题/路径关键字与任务目标、修改范围匹配
- `lattice context --task <id>` 的语义关联推荐
- 认知类 spec 默认应读（架构/模块/方法论/流程地图）
- 约束类 spec 按本次修改范围决定

### 如何读

1. **必须 `read_file` 读正文**——看到标题不等于了解内容
2. **验证时效性**——如发现 spec 与当前代码矛盾，做交叉检查，必要时询问用户
3. **提炼影响**——对当前任务的具体约束和可复用经验

### 关联

实施时参照了某 spec → `lattice task ref-spec <task-id> <spec-name>`。

## 写 spec（沉淀判定 + 写入流程）

### 何时写

spec 沉淀不是只在归档时才做的事。**在多轮对话完成任务的过程中，任何时候发现可沉淀的内容，都应该立即行动。**

| 档位 | 条件 | 示例 |
|---|---|---|
| **必须写** | 用户显式指示行为规则 / 用户主动给出项目认知 | “只改源码不碰产物” / “这个数据流是…” |
| **建议写** | 满足核心判定问题（下次还需要这条信息吗？） | 迁移方法论 / 模块职责 / 踩坑记录 / API 映射 |
| **不写** | 一次性需求 / 未验证猜测 / 纯任务级细节 / 单次 workaround | 临时 bug 排查过程 |

**AI 主动行为**：发现符合“必须写”或“建议写”条件的内容时，主动询问用户是否沉淀为 spec。不要放到归档时才想起来，也不要默默跳过。

### 写入流程

按顺序执行，不跳步：

**1. 前置采集**

```bash
lattice context
lattice spec list --scope <层级>
# 有活跃任务时：
lattice task info <task-id>       # → read_file prd.md
lattice task progress <task-id>
# 回顾对话中的最终结论
```

**2. 查已有 spec**

```bash
lattice spec template list          # 有无可参考模板
lattice spec show <相关文件>         # 优先补充而非重复创建
```

**3. 冲突检测**

读取同层级及上下层级已有 spec，判断语义矛盾（不仅看同名）。发现冲突告知用户。

**4. 写入**

- 一个文件只聚焦一个主题
- 写具体可执行的内容，不写空泛口号
- **包含完整操作步骤和经验细节**——映射关系、踩坑点、边界处理、失败方案及否决理由
- user/global 必须带 `## 适用范围`

**5. 二次审阅**

对照 PRD + progress + 对话检查：有无遗漏、有无写成废弃方案、有无应一并沉淀但被忽略的内容。

**6. 索引更新**

```bash
lattice rag update
```

