# Spec 工作流

spec 概念/层级/精读/沉淀判定/写入流程。其他文档涉及 spec 时引用本文，不复述。

## spec 是什么

spec 记录对理解项目、完成任务有益的可复用信息。核心判定：**下次进入这个项目还需要这条信息吗？**

内容范围：行为约束 · 项目认知（架构/模块/领域概念）· 流程范式 · 经验细节 · 试错积累

spec 不是永远正确——发现与代码矛盾时交叉验证，必要时询问用户。

## 层级

```
项目级 > [父项目级 > 祖先项目级 >] 用户级 > 全局级
```

| 层级 | 路径 | 适用 |
|---|---|---|
| 项目级 | `~/.lattice/users/<u>/projects/<id>/spec/` | 当前项目 |
| 用户级 | `~/.lattice/users/<u>/spec/` | 跨项目可复用 |
| 全局级 | `~/.lattice/spec/` | 多用户多项目 |

冲突：近覆盖远，项目级最优先；同名冲突必须告知用户。user/global 必须含 `## 适用范围`。

## 按任务主题精读相关 spec

> 委派：spec 多或涉及不熟悉模块时必须委派 `lattice-spec-digest` subagent（不支持时退化串行）。subagent 读取并筛选相关 spec，返回目录；**主线凭目录 Read 全文，并可自主调用 `ltc search` / `ltc context` 补全信息**。

> **动态持续过程，非一次性**。起手读一批后，任务推进中涉及新模块/新概念时持续补读。宁多勿少。

### 如何选

**第一步**：从 `ltc context` 输出的标题+描述选读。不确定是否相关 → 读而非跳过。

**第二步**：`ltc search "<任务关键词>" --json` 语义补漏（发现标题不直观但内容相关的 spec + 历史任务 PRD）。

### 如何读

1. `read_file` 读完整正文（禁止部分读取）
2. 验证时效性（与代码矛盾时交叉检查）
3. 提炼对当前任务的约束和经验

### 关联

参照了某 spec → `ltc task ref-spec <task-id> <spec-name>`

## 沉淀判定

任何时候发现可沉淀内容都应立即行动。

| 档位 | 条件 |
|---|---|
| **必须写** | 用户显式指示行为规则 / 用户主动给出项目认知 |
| **建议写** | 满足核心判定（下次还需要？） |
| **不写** | 一次性需求 / 未验证猜测 / 纯任务级细节 |

### 基于 checkpoint 类型

| 类型 | 沉淀条件 | 目标层级 |
|---|---|---|
| `correction` | 反映长期行为规范（含试错经验）→ 强制 | 项目级或用户级 |
| `constraint` | ≥2 任务复现 → 强制；首次按"建议写" | 项目级→复现升用户级 |
| `context` | 业务领域/架构背景 → 认知类 spec | 项目级 |
| `assumption` | 被确认且通用 → 上升为规则 | 视内容 |

`issue` + 解决方案 = 试错积累类 spec。发现符合条件 → 主动询问用户。

## 写入流程

1. 前置：`ltc context` + `ltc spec list --scope <层级>` + 回顾对话
2. 查已有：`ltc spec show <相关>`，优先补充非重复创建
3. 冲突检测：同层及上下层语义矛盾 → 告知用户
4. 写入：一文件一主题 · 具体可执行 · user/global 带 `## 适用范围` · **敏感信息不写正文**（写 `~/.lattice/.cache/sensitive/`，正文引路径）
5. 元数据：编辑正文后 `ltc spec migrate`；仅改 frontmatter 用 `ltc spec set`
6. 二次审阅
7. `ltc rag update`

## 模板命令

```bash
ltc spec template list / apply <name> / pull <repo> / sync / registry list
```
