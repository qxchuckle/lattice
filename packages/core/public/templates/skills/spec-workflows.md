# Spec 工作流

本文件用于处理 spec 的读取、冲突判断、模板应用和规则沉淀。

## 层级优先级

spec 读取顺序（基本三层）：

1. `~/.lattice/users/<username>/projects/<id>/spec/`（项目级）
2. `~/.lattice/users/<username>/spec/`（用户级）
3. `~/.lattice/spec/`（全局级）

**嵌套项目场景扩展为五层**（当子项目位于父项目目录内时自动生效）：

1. 当前项目 spec（最高优先）
2. 直接父级项目 spec
3. 更远祖先项目 spec
4. 用户级 spec
5. 全局级 spec（最低优先）

同名文件冲突时：

- 项目级优先
- 祖先 spec 按距离覆盖（近覆盖远）
- 用户级覆盖全局级
- 需要明确告诉用户覆盖关系，而不是静默忽略

## 先读再改

涉及规范、约定、历史方案时，优先运行：

```bash
lattice context
lattice spec list
lattice spec show <相关文件>
lattice spec show <相关文件> --user <username> --detail  # 查看其他用户为同一项目定义的 spec
```

> `--user` 允许查看其他用户在同一项目中定义的 spec，有助于了解团队不同成员的规范理解。不存在的用户名会报错并列出可用用户。

**不仅在写入前要读，任务起手时也要读**：spec 同时是行为约束和项目认知经验。任务开始后，必须按当前任务主题挑选与之相关的 spec（特别是项目结构 / 模块职责 / 领域概念 / 方法论等**认知类**）做 read_file 精读，不能只看 `lattice context` 输出的 spec 标题列表就以为已经了解。详见 `task-workflows.md` 的"按任务主题选读相关 spec"小节。

## 冲突检测原则

`lattice spec conflicts` 只检测同名文件冲突，但真正的冲突往往是语义层面的。写入新 spec 前必须：

1. 读取目标层级及其他层级的相关 spec 内容
2. 判断新规则是否与已有规则存在矛盾（而不仅仅是同名）
3. 如有冲突，明确告知用户：冲突点、影响范围、建议处理方式（保留特例 / 合并规则 / 删除冗余）

## 何时写入 spec

> **核心判定**：下次有人 / AI 进入这个项目，是否还需要这条信息？如果需要，且它在该任务结束后仍然稳定有效，就应沉淀。

spec 同时承担两种职能：**约束 AI 行为的规范** + **AI 认识项目的稳定经验**。两类内容都值得沉淀。

### 类别 A：行为约束类（应该怎么做 / 不要怎么做）

适合沉淀：

- 多次重复出现的编码约定、命名规则、提交流程
- 架构边界、目录规范、依赖方向、模块协作规则
- 已验证有效的排错或交付流程
- 用户显式给出的"必须 X / 不能 Y"规则

### 类别 B：项目认知经验类（这是什么 / 为什么这样 / 怎么处理）

适合沉淀（**不要因为它"不像规范"就丢弃**）：

- **项目结构与模块职责**：哪个包做什么、目录约定、关键代码位置
- **领域概念与术语**：业务术语、数据结构语义、状态机定义
- **设计动机与历史决策**：为什么这么设计、否决了哪些方案、tradeoff
- **可复用方法论**：解决某类问题的标准思路（迁移流程、调试套路、定位手段、排查清单）
- **关键流程地图**：模块入口、调用链概览、扩展点与适配方式

### 不适合沉淀

- 一次性临时需求 / 单次 bug 排查过程
- 只对当前任务有效的 workaround
- 仍未确认的猜测 / 未验证的方案
- 纯任务级细节（应进 PRD / progress，而不是 spec）

## 如何选择写入层级

- 只对当前项目长期有效：项目级
- 跨多个项目可复用，但仍属于当前用户/团队习惯：用户级
- 对多用户、多项目都成立的默认规则：全局级

## user/global 级 spec 必须声明适用范围

用户级和全局级 spec 的作用范围远超单个项目。**必须**包含 `## 适用范围` 段落，明确声明：

- 该规范适用于哪些项目/语言/框架
- 不适用于哪些情况（如果有）
- 如何判断当前项目是否落入本规范的范围

**为什么：** 缺失适用范围会导致 AI 在聚合上下文时盲目将所有 user/global spec 应用到当前项目，产生误导。

**project 级 spec 不需要**，因为它天然只对当前项目生效。

`lattice spec list` 会对缺失适用范围的 user/global spec 输出警告。

**参考示例：**

```markdown
## 适用范围

- 适用于所有前端项目（React / Vue / 纯 TS 库等），不限定特定工作空间
- 本规范为通用导入约定，如果某个项目有更严格的项目级规则，以项目级为准
```

或限定为特定项目族：

```markdown
## 适用范围（重要）

> **本规范仅适用于 HBOS 项目族**，不要应用到非 HBOS 的前端项目。

具体覆盖以下工作目录下的仓库：...
```

## 模板与新建 spec

新建 spec 前应先检查是否有可参考的模板：

```bash
lattice spec template list
```

如果有匹配的模板（如 frontend、backend、api、conventions 等），参考其文件组织和内容结构再写入。不必完全照搬，但应保持一致性。

## 模板相关命令

```bash
lattice spec template list
lattice spec template apply <name>
lattice spec template pull <repo>
lattice spec template sync
lattice spec template registry list
```

## 输出要求

- 说明你读取或更新了哪个 spec 文件
- 提炼核心规则，不要只罗列文件名
- 如果发现更合适的层级，明确给出建议

## 索引更新

新建或修改 spec 文件后，应运行：

```bash
lattice rag update
```

确保新写入的规范能被 `lattice search` 检索到。如果 `rag update` 报错，降级使用 `lattice rag rebuild`。
