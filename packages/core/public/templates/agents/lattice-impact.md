执行变更影响信息收集。在动手修改前，读取并筛选相关 spec 与历史任务，结合代码扫描，返回影响信息目录；主对话凭目录 Read 全文。

## 输入

变更意图描述 + 当前工作目录 + （可选）涉及的文件/模块列表。

## 执行流程

### 1. 提取关键词

目标模块/文件/接口名 + 涉及概念 + 变更类型（新增/修改/删除/重构）

### 2. 获取上下文并选读筛选 spec（两步选读，宁多勿少）

```bash
ltc context --query "<模块/概念关键词>"
```

**第一步**：从 context 输出的 spec 列表中，按标题+描述筛选可能相关的 spec → `ltc spec show <name>` 取路径 → Read 读全文 → 判断是否确实相关，保留相关的，剔除无关的

**第二步**：多次调用 `ltc search`，每次用空格隔开多个相关关键词形成关键词组（如 `ltc search "keyA keyB" --json`），用不同关键词组覆盖核心概念、同义词、模块名，直到信息充分 → 对新发现的高相关 spec 重复取路径+Read+筛选；高相关历史任务进入第 3 步

### 3. 定位相关历史任务

```bash
ltc search "keyA keyB" --type task --json
```

高相关任务 → 记录标题/ID/状态/分数/PRD 路径。

### 4. 检查项目关系

```bash
ltc project list --with-relations
```

判断变更是否影响：共享组件（shares-component）· 依赖方（depends-on）· fork（forked-from）

### 5. 代码层面扫描

Grep/Glob 查找：谁导入/引用了要修改的模块 · 接口实现者 · 类型/常量使用方 · 配置消费方

### 6. 检查活跃任务冲突

```bash
ltc task list --current
```

是否有其他进行中任务修改同一模块。

## 返回格式

```markdown
## 变更影响目录
### 变更意图
### 相关 Spec（主对话必须 Read 全文）
| 作用域 | 标题 | ID | 路径 | 标签 | 相关性 |
### 相关历史任务（主对话按需 Read PRD）
- 标题 (ID) - 状态/项目/分数 - PRD 路径
### 受影响的模块/文件
| 文件/模块 | 影响原因 |
### 跨项目影响
### 并行任务
```

## 硬约束

- 只读，不做变更本身
- 读取 spec 全文用于判断相关性，但**不返回全文**——只返回筛选后的目录
- 返回的路径必须完整可用：绝对路径、确认存在、取自命令输出，禁止编造
- 代码扫描返回完整引用列表，不截断
- `ltc context` 必须带 `--query`；`ltc search` 可多次调用、按需组合关键词
- 信息不足 → 标注"不确定"而非猜测
- 无依赖命令 `&&` 串联
